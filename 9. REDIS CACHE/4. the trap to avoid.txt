But now there's a trap hiding in what you just built. Let me show you.

You noticed the sticky note lasts 5 minutes. So picture this:

1. Dashboard loads → note written with London at 24°C.
2. 30 seconds later, your Celery worker fetches fresh data — London is now 26°C, saved to the DB.
3. Someone loads the dashboard → the receptionist glances at the note → still says 24°C → serves the stale number.

For the next 4.5 minutes, new data is sitting in the database but nobody sees it. The note is lying. This is the #1 silent cache bug — no error, no crash, just quietly wrong numbers.

The fix is the other half of the receptionist story: when the menu changes, tear off the note. In code that's cache.delete.

---
Now here's the single most important detail — and it's a place people get silently burned:

cache.set('dashboard_data', context, 300)   # in views.py  — writes THIS note
cache.delete('dashboard_data')               # in tasks.py  — tears THIS note

The key string 'dashboard_data' must match exactly in both files. The note in the view and the note in the task are the same physical note only because the name matches character-for-character. Typo it as 'dashboard-data' in one place and you've torn a different note — the real one stays stale, and there's no error to tell you. That string is a contract between two files, exactly like the context keys were a contract between view and template (your NOTES_05 "bag").

---
The full loop now:

Worker fetches new data → bulk_create → cache.delete('dashboard_data')  [note torn]
                                              │
Next dashboard load → cache.get('dashboard_data') → None (MISS)
                                              │
                                  fresh DB query → cache.set(...)  [new note written]

Stale data is now impossible — the moment new rows arrive, the old note is gone.
