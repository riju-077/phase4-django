---
1. New import at the top
from django.core.cache import cache
This is the phone. cache is Django's one object for talking to Redis — .get, .set, .delete all hang off it. Django already knows which Redis to call because of your CACHES = {...} block in settings.py (the saved phone number).

---
2. The flow flipped: check the note FIRST
context = cache.get('dashboard_data')
Before touching the database, the view glances at the sticky note named 'dashboard_data'. Two outcomes:
- Note is filled → context is your data → skip everything, go straight to render.
- Note is blank → cache.get returns None → we fall into the if.

---
3. The if context is None: block = the cache miss path
if context is None:
    latest = WeatherReading.objects.order_by('-recorded_at').first()
    readings = list(WeatherReading.objects.order_by('-recorded_at')[:24])
    context = {'latest': latest, 'readings': readings}
    cache.set('dashboard_data', context, 300)
This is your old code, unchanged — the two DB queries and the context bag. It now only runs when the note is blank. The last line writes the finished bag onto the note for 300 seconds.

---
4. One small but important change: list(...)

Your original was:
readings = WeatherReading.objects.order_by('-recorded_at')[:24]
I wrapped it in list(...). Here's why: a Django queryset is lazy — it's a promise to hit the DB later, not the actual rows. You can't store a promise on a sticky note; the note needs the real answers. list(...) forces the query to run now and gives us concrete rows to cache. (latest is already concrete because .first() runs immediately.)

---
5. Render happens once, at the bottom
return render(request, 'weather/dashboard.html', context)
Notice it's outside the if. Whether context came from the note or from a fresh DB walk, the render line is the same. That's the elegance — the rest of the view doesn't care where the data came from.
