That's it. A Dockerfile is just the setup steps a human would do — written down so a machine repeats them identically, forever.

Every manual step from that painful 14-step story? Steps 1–6 above. That's the same work, automated once.

---
Now the one idea worth actually understanding: why deps are copied before code.

Docker builds in layers, and it caches each one. If a layer's inputs haven't changed, it reuses it instead of redoing it.

Look at the order:

COPY pyproject.toml uv.lock ./   ← changes rarely
RUN uv sync --locked             ← slow (37 packages)
COPY . .                         ← changes constantly

You edit views.py and rebuild. Docker checks: did pyproject.toml change? No → reuses the installed packages. Only redoes the last step.

Rebuild: a couple of seconds.

Now flip it — code first, deps after. You edit views.py, and everything downstream is invalidated. All 37 packages download again. Every single build.

Same instructions. Different order. Seconds vs minutes, forever.

▎ Put what rarely changes first. Put what always changes last.

That's the whole art of writing Dockerfiles, honestly.
