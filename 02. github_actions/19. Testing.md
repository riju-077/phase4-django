# NOTES 10 — Tests (the little robots that check your app)

**Date:** 2026-07-18

## What is a test, really?

A test is a **tiny robot that uses your app for you and checks the answer.**

You already did this **by hand** many times:

- You added a user in the admin → then checked "did an OTPCode appear?"
- You opened the dashboard → then checked "did the page load?"

A test just **writes that check down once**, so a robot can repeat it forever.

---

## Why bother writing tests?

Because apps grow. One day you'll change some code and **accidentally break**
something you built weeks ago — and you won't notice, because nobody re-checks
everything by hand every time.

The robot re-checks **everything, in one second.** If you broke the OTP signal,
the robot **fails loudly** and tells you *before* your users hit it.

Even better: **CI runs these robots automatically** on every push. That's the real
payoff — your safety net runs itself.

---

## One thing that makes tests safe to run

`python manage.py test` builds a **separate, throwaway test database**, runs all
the checks inside it, then deletes it. It **never touches your real data.**
So running tests is 100% safe.

---

## The 4 robots we wrote (in `weather/tests.py`)

Each one is a real check you'd otherwise do by hand:

1. **Make a user → an OTPCode appears** (your "sky + DBeaver" check, automated)
2. **Generate an OTP code → it verifies as correct**
3. **Type a wrong code → it gets rejected**
4. **Open the dashboard page → it loads (doesn't crash)**

The full file:

```python
from django.test import TestCase
from django.urls import reverse
from django.contrib.auth.models import User


class SignalTest(TestCase):
    """Check: making a new user automatically creates an OTPCode for them."""

    def test_creating_a_user_auto_creates_an_otpcode(self):
        user = User.objects.create_user(username="testuser", password="pw123456")
        self.assertTrue(hasattr(user, "otp"))     # an OTPCode got attached
        self.assertIsNotNone(user.otp.secret)     # and it has a secret


class OTPModelTest(TestCase):
    """Check: the OTP code we generate actually verifies correctly."""

    def test_a_freshly_generated_code_is_valid(self):
        user = User.objects.create_user(username="otpuser", password="pw123456")
        code = user.otp.generate_otp()            # make a code
        self.assertTrue(user.otp.verify_otp(code))  # the same code should pass

    def test_a_wrong_code_is_rejected(self):
        user = User.objects.create_user(username="otpuser2", password="pw123456")
        self.assertFalse(user.otp.verify_otp("000000"))  # a bad code should fail


class DashboardViewTest(TestCase):
    """Check: the dashboard page opens without crashing."""

    def test_dashboard_page_loads(self):
        response = self.client.get(reverse("dashboard"))
        self.assertEqual(response.status_code, 200)   # 200 = page loaded OK
```

## How to read the test code (plain English)

```python
class SignalTest(TestCase):          # a GROUP of related checks
    def test_creating_a_user_...:    # ONE check — the name MUST start with "test_"
```

- **`TestCase`** = Django's test base. It's what makes the throwaway test database.
- **method name starts with `test_`** = that's how Django knows to run it.

Inside a check there's a simple rhythm — **do something, then check the result**:

```python
user = User.objects.create_user(...)       # DO: make a user (fires your signal)
self.assertTrue(hasattr(user, "otp"))      # CHECK: did an OTPCode get attached?
```

The `assert...` lines are the actual checks:

| Line | Means |
|------|-------|
| `assertTrue(x)`      | x must be true, or FAIL |
| `assertFalse(x)`     | x must be false, or FAIL |
| `assertIsNotNone(x)` | x must not be empty, or FAIL |
| `assertEqual(a, b)`  | a must equal b, or FAIL |

`self.client.get(...)` is a **fake browser** Django gives you — it "visits" a page
inside the test, so you can check the response without opening a real browser.

## Is this a "quality engineer" thing?

**Yes.** Testing is the heart of **QA / Quality Engineering** (also called SDET —
Software Development Engineer in Test). That's a whole career built around writing
these robots.

But the key point: **normal developers write tests too.** It is not a separate job
you hand off — it is a **core engineering skill.** A developer who writes tests is
simply a good developer. QA engineers go *deeper* (huge test systems, tricky edge
cases, automation frameworks), but the basic idea here is the same one they use.

So writing these 4 tests = real quality-engineering work.

---

## How to run them

```
python manage.py test weather
```

Postgres must be up (`docker compose ps` to confirm). You want to see:

```
Ran 4 tests in 0.0XXs
OK
```

- **OK** = all robots passed.
- **FAILED** = a robot found a problem — read which test failed and why, then fix.
