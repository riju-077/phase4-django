# NOTES 08 — Config Cleanup (secrets out of the code)

**Date:** 2026-07-18

## The one-line goal

Three important settings were **hard-coded** inside `settings.py`.
We moved them **out of the code** and into the `.env` file (which git ignores).

Why this matters: `settings.py` is committed to GitHub. Anything written
there is public to anyone who can see the repo. Secrets must never live in
committed code. `.env` is git-ignored, so it stays private on your machine.

Files touched: **3** — `settings.py`, `.env`, `.github/workflows/ci.yml`.

---

## The pattern (you already know this)

Same trick you used for the database:

```python
config("SOME_NAME")   # go read SOME_NAME from the .env file
```

The only new thing today: **what happens when the value is missing** — and that
answer is different for each of the three settings. That is the real lesson.

---

## File 1 — `settings.py` (the main change)

### 1a. SECRET_KEY

**BEFORE**
```python
SECRET_KEY = "django-insecure-nmtvt-ejpwligkgro30^i)t!poj%1gry+bg)3x1^%41#3!+%-+"
```

**AFTER**
```python
SECRET_KEY = config("SECRET_KEY")
```

**Why no "default" value?**
A secret key has **no safe backup value**. If we gave it a default, that default
would itself be a guessable key — which is the exact danger we are trying to
avoid. So we do the opposite: if `SECRET_KEY` is missing, the app should
**crash immediately**. A loud crash is better than quietly running on a weak key.

> Rule of thumb: *if there is no safe fallback, fail loud.*
> (Same idea as `uv sync --locked` from the CI day — crash on a problem, don't hide it.)

---

### 1b. DEBUG

**BEFORE**
```python
DEBUG = True
```

**AFTER**
```python
DEBUG = config("DEBUG", default=False, cast=bool)
```

**Why give this one a default of `False`?**
Here a safe backup value **does** exist: **off**.
`DEBUG = True` shows full error pages (with code and settings) to whoever visits.
Great on your laptop, dangerous on a real server. So if a server ever forgets to
set `DEBUG`, we want it to fall back to the **safe** value — off — automatically.

**Why `cast=bool`?**
Everything in a `.env` file is text. `DEBUG=True` is the *string* `"True"`, not the
real Python `True`. `cast=bool` converts that text into a proper boolean.

> Rule of thumb: *if a safe fallback exists, default to it.*

---

### 1c. ALLOWED_HOSTS

**BEFORE**
```python
ALLOWED_HOSTS = []
```

**AFTER**
```python
ALLOWED_HOSTS = config("ALLOWED_HOSTS", default="127.0.0.1,localhost", cast=Csv())
```

**What is this setting?**
A list of web addresses your site is allowed to answer on. It's a safety check
that only turns on when `DEBUG=False`. While `DEBUG=True`, Django ignores it, so
it does nothing on your laptop right now — we're just setting it up for later.

**Why `Csv()`?**
A `.env` value is a single line of text. `Csv()` splits a comma-separated line
into a real Python list:

```
"127.0.0.1,localhost"   →   ['127.0.0.1', 'localhost']
```

This is the one new import at the top of the file:

```python
# BEFORE
from decouple import config
# AFTER
from decouple import config, Csv
```

---

### The big takeaway from `settings.py`

Look at the two opposite choices side by side:

| Setting     | Missing value → | Why |
|-------------|-----------------|-----|
| SECRET_KEY  | **crash**       | no safe fallback exists; a default key IS the danger |
| DEBUG       | **default False** | a safe fallback exists (off); use it |

The decision each time = **"Is there a safe fallback value?"**
Yes → default to it. No → fail loud.

## File 2 — `.env` (where the real values now live)

**ADDED at the top:**
```ini
# Django core — moved OUT of settings.py so secrets never sit in committed code
SECRET_KEY=django-insecure-nmtvt-ejpwligkgro30^i)t!poj%1gry+bg)3x1^%41#3!+%-+
DEBUG=True
```

- The `SECRET_KEY` is the **exact same value** as before — we only *moved* it.
- `DEBUG=True` because this is your development machine.
- `.env` is git-ignored, so these never reach GitHub. That's the whole point:
  the value went from **committed code** → **private local file**.

---

## File 3 — `.github/workflows/ci.yml` (the ripple)

Because `SECRET_KEY = config("SECRET_KEY")` has **no default**, it will crash if
the value is missing. CI (GitHub's test machine) has **no `.env` file** — so CI
would now fail with `SECRET_KEY not found`.

Fix: give CI its own throwaway value, right next to the fake Postgres ones
already there.

**BEFORE**
```yaml
    env:
      POSTGRES_DB: dashboard
      POSTGRES_USER: dashboard
      POSTGRES_PASSWORD: postgres
```

**AFTER**
```yaml
    env:
      SECRET_KEY: ci-throwaway-secret-guards-nothing
      POSTGRES_DB: dashboard
      POSTGRES_USER: dashboard
      POSTGRES_PASSWORD: postgres
```

This value is **fake on purpose**. CI only builds a throwaway machine to run
`manage.py check` for a few seconds — it protects nothing real. It just lets
`settings.py` finish loading.

`DEBUG` and `ALLOWED_HOSTS` need **no** CI line — they both have safe defaults,
so they load fine without one.

---

## Two honest caveats (not done yet — just flagged)

1. **The key still says `django-insecure-`.** That prefix is Django's own label
   for "auto-generated, don't use in production." Moving it to `.env` did NOT make
   it production-grade — and it's already sitting in old git history. When you do a
   **real** deploy, generate a brand-new key. Today we only got it out of the
   current source code.

2. **Do NOT set `DEBUG=False` in the container yet.** When DEBUG turns off, Django
   stops serving CSS/static files itself, so the admin page loses all its styling.
   Fixing that needs a tool called **whitenoise** — that's a separate step, coming
   next.

---

## How to verify

```bash
python manage.py check
```

If it passes, `settings.py` still loads correctly with the values now coming from
`.env` instead of from hard-coded lines.
