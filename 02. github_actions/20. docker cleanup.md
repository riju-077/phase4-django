# NOTES 11 — Cleaning up Docker to free your laptop

**Date:** 2026-07-18

## The big idea (read this first)

GitHub stores your **recipe** — the code, the `Dockerfile`, the `docker-compose.yml`.
It does **NOT** store your running boxes, and it does **NOT** store your **data**.

Think of it like cooking:

- **GitHub** = the written recipe.
- **Images** = the frozen meal kit built from the recipe.
- **Containers** = the meal actually cooking on the stove.
- **Volumes** = the ingredients *you* added while cooking — your real data.

You can always re-print the recipe and cook again. But whatever you *made and
stored* (the data) is only in the volume. GitHub never had it.

---

## What lives where

| Thing in Docker Desktop | Is it in GitHub? | Safe to delete? | If deleted... |
|---|---|---|---|
| **Containers** (web, worker, beat, db, redis) | No — but rebuildable from code | ✅ Yes | rebuilt in seconds from the recipe |
| **Images** (dashboard, postgres, redis, python) | No — but rebuildable | ✅ Yes | rebuilt/re-downloaded on next `up --build` |
| **Volumes** (`pgdata`, `redisdata`) = your DATA | ❌ **NO** | ⚠️ Only if OK losing data | **data gone for good** |

---

## What exactly do you LOSE if you delete the volumes?

The volumes hold everything **inside** your database, which GitHub never had:

- All the **weather readings** you fetched.
- Your **superuser** login (e.g. "ben") and any other users.
- The **OTPCode** records.
- Anything else stored in Postgres/Redis.

**For this learning project, that's fine** — the data is disposable. You can make a
new superuser and re-fetch weather anytime. Just go in **knowing** the next start
will have an **empty database**.

You do NOT lose:

- Your code (it's on GitHub).
- Your ability to rebuild and run the whole thing again.

---

## `down` vs `down -v` — the one command difference that matters

```
docker compose down        → removes containers, KEEPS volumes (data SAFE)
docker compose down -v      → removes containers AND volumes (data DELETED)
```

The `-v` is the whole difference. During normal daily work you AVOID `-v` (so you
don't nuke your data by accident). But right now you *want* to wipe everything to
free space — so `-v` is exactly the flag you want. Same command, opposite intent.

---

## Commands to free your laptop

Run these **from inside the `phase4` folder** (the one with `docker-compose.yml`).
Pick the level you want:

### Level 1 — just stop it (keeps everything, frees no disk)
```
docker compose stop
```
Boxes turn off. Nothing deleted. Use this between work sessions.

### Level 2 — remove containers, KEEP your data
```
docker compose down
```
Containers gone, but volumes (data) stay. Next `up` still has your weather + users.

### Level 3 — remove containers AND data (this project)
```
docker compose down -v
```
Containers + volumes gone. Next start = empty database.

### Level 4 — remove containers, data, AND images (max space, this project)
```
docker compose down -v --rmi all
```
This is the **full wipe for this project**: containers, volumes, and every image it
used (your `dashboard` image plus the pulled `postgres` and `redis` images).
This is the one to run if your goal is "reclaim the most space from Phase 4."

---

## Doing it in the Docker Desktop app instead (clicks, not commands)

If you prefer the GUI:

1. **Containers** tab → select the `phase4_*` containers → **Delete**.
2. **Volumes** tab → select `pgdata` and `redisdata` → **Delete**. ⚠️ *This is the
   data — deleting here is the point of no return.*
3. **Images** tab → delete the `dashboard` image (and `postgres` / `redis` if you
   want their space back too).

Same result as `docker compose down -v --rmi all`, just by clicking.

---

## ⚠️ One command to AVOID unless you really mean it

```
docker system prune -a --volumes
```
This wipes **everything not currently running across your WHOLE machine** — not just
Phase 4, but every other Docker project too. Only use it if you truly want a
clean-slate Docker with nothing left. For just freeing Phase 4, use Level 4 above.

---

## How to bring it ALL back later

Nothing is lost that matters — here's the full restore:

```
git clone <your-repo-url>
cd phase4
docker compose up -d --build
```
That rebuilds the images and starts all 5 containers from the recipe.

The database starts **empty**, so finish with:
```
docker compose exec web python manage.py createsuperuser
```
then open the app and hit **/fetch/** to pull fresh weather data.

---

## Small correction on "others can just use it"

Right now, other people use your project by **cloning the code** and running
`docker compose up --build` — which builds the image on **their** machine from your
`Dockerfile`. They cannot yet "pull your ready-made image" directly.

That "grab the prebuilt image off a shelf" ability is the **ghcr.io step we
deferred** — once that's done, someone could pull your built image without building
it themselves. Until then: clone the code, build locally.
