# NOTES 07 — CI/CD Part 1: Getting the build green

Date: 2026-07-16

## What CI actually is

Every time you `git push`, GitHub hands you a **brand-new, empty computer**.

1. It downloads your code **from GitHub** onto that fresh machine
2. It runs the commands listed in `.github/workflows/ci.yml`
3. It throws the machine away

That's it. That's all CI is.

The key phrase is **"from GitHub"**. Only what you pushed exists on that machine.
Anything git-ignored (like `.env`) is **not there**.

## Bug 1 — `ModuleNotFoundError: No module named 'celery'`

### How we found it
Read the **last line** of the traceback in the Actions log. Not the middle. The last line.

The log also showed:
```
Collecting django (from -r requirements.txt (line 1))
Collecting djangorestframework (from -r requirements.txt (line 2))
```
Line 1, line 2 — that was the **whole file**.

### The cause

| File | Contents | Who updates it |
|---|---|---|
| `requirements.txt` | 2 packages | **nobody** — written once, months ago |
| `pyproject.toml` | 11 direct packages (37 resolved) | `uv add`, every time |

Every `uv add celery` / `uv add httpx` / `uv add pyotp` wrote to **`pyproject.toml`**.
It **never touched `requirements.txt`**.

CI read `requirements.txt`. Installed 2 packages. Then Python opened
`dashboard/__init__.py`, hit `from .celery import app`, and died.

My laptop never noticed because my laptop uses the real list.

### Lesson 2

> **Two sources of truth means one of them is lying — you just don't know which yet.**
> Nothing forces two lists to agree, so one will rot. Keep one list.

## Fix for Bug 1 — make CI use `uv`

Chose: delete `requirements.txt` entirely, make CI use the same tool the laptop uses.
(The alternative — regenerate `requirements.txt` from `pyproject.toml` each time —
still leaves two files and a step you must remember. You will forget.)

Docs used: <https://docs.astral.sh/uv/guides/integration/github/>

### Before
```yaml
- name: Set up Python 3.12
  uses: actions/setup-python@v5
  with:
    python-version: '3.12'
    cache: 'pip'

- name: Install dependencies
  run: |
    python -m pip install --upgrade pip
    pip install -r requirements.txt

- name: Run Django system check
  run: python manage.py check
```

### After
```yaml
- name: Install uv
  uses: astral-sh/setup-uv@v5
  with:
    enable-cache: true

- name: Install dependencies
  run: uv sync --locked

- name: Run Django system check
  run: uv run python manage.py check
```

Plus: `git rm requirements.txt`

### What each piece does

- **`astral-sh/setup-uv@v5`** — installs uv on the CI machine. `enable-cache: true`
  reuses downloaded packages between runs so CI stays fast.
- **no more `setup-python`** — uv reads your `.python-version` file (`3.12`) and
  installs that Python itself. Fewer moving parts.
- **`uv sync --locked`** — installs exactly what `uv.lock` pins, **and fails loudly
  if `uv.lock` disagrees with `pyproject.toml`.**
- **`uv run python ...`** — runs the command inside the environment uv just built.

### Lesson 3 — the important one

`--locked` doesn't just install. It **checks for drift and fails**.

> Don't just fix the bug. **Remove the possibility of the bug.**

17 days were lost to two files silently disagreeing. `--locked` turns that exact
silent lie into a loud failure. That's the upgrade — not the fix itself.

### Verify locally before pushing
```powershell
uv sync --locked
```
Healthy output:
```
Resolved 37 packages in 5ms
Audited 36 packages in 0.22ms
```
37 packages vs `requirements.txt`'s 2 — because your 11 direct deps have their own
deps, and those have deps. **`uv.lock` records the full real tree.**

## Bug 2 — `decouple.UndefinedValueError: POSTGRES_DB not found`

Predicted this one before pushing. It was hiding **behind** Bug 1 — the celery crash
happened at import, before Django ever read `DATABASES`.

> **Two bugs stacked. Fix the top one, the next steps forward.**
> A *different* error is a **win** — it proves the first bug is genuinely dead.

The error message handed over the answer:
```
POSTGRES_DB not found. Declare it as envvar or define a default value.
```
Good libraries tell you your options. **Read the message, don't skim it.**

### The cause
`.env` is git-ignored (correct — it holds the real DB password), so it never reaches
GitHub. `config("POSTGRES_DB")` had nothing to find.

### The realisation

> **CI doesn't need your secrets. It needs its own.**

Why would CI need *your* database password? It's never touching your database —
your Postgres is a container on your laptop; GitHub can't even reach it.
CI just needs *some* values so `settings.py` can finish loading.

### Fix — job-level `env:` block in `ci.yml`
```yaml
jobs:
  check:
    runs-on: ubuntu-latest

    env:
      POSTGRES_DB: dashboard
      POSTGRES_USER: dashboard
      POSTGRES_PASSWORD: postgres

    steps:
      ...
```

### Lesson 4 — the big one

> **Config isn't secret because of its NAME. It's secret because of what it PROTECTS.**

- `POSTGRES_PASSWORD` in CI → guards a throwaway DB that lives ~15 seconds inside
  GitHub's machine. Protects **nothing**. A prop. Safe in plain text in the repo.
- `.env` on the laptop → guards **real data**. Stays git-ignored. Forever.
- **Real** secrets (ghcr.io token, API keys) → **GitHub Secrets**, never in the
  workflow file. That comes when we push the image.

### Why only those 3 variables?

Look at `settings.py`:
```python
"NAME":     config("POSTGRES_DB"),                        # no default → crashes
"USER":     config("POSTGRES_USER"),                      # no default → crashes
"PASSWORD": config("POSTGRES_PASSWORD"),                  # no default → crashes
"HOST":     config("POSTGRES_HOST", default="127.0.0.1"), # has default → fine
"PORT":     config("POSTGRES_PORT", default="5432"),      # has default → fine
```

**You already solved this weeks ago** — for `HOST` and `PORT`, with `default=`.
You just didn't do it for the other three. `REDIS_URL` has a default too, so it
survived CI untouched.

## The method (worth more than the fix)

I predicted the `.env` error first. **I was wrong** — reality gave us `celery`.
That's exactly why we looked instead of trusting the prediction.

The loop:

```
1. LOOK        → open the actual log, don't theorise
2. READ        → last line of the traceback first, then work up
3. FIX ONE     → one change only, never stack two guesses
4. OBSERVE     → push, watch, read the NEW error
5. repeat
```

Same bottom-up traceback reading as Phase 3 — just pointed at a machine in a data
centre instead of a laptop.
