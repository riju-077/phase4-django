2. The discovery pattern — you've got the right mindset 👏

Exactly right: "pretend I know nothing, first ask the database what it contains." That's how you onboard onto any unfamiliar project. Never assume — ask the DB.

Query 1 — "what tables even exist here?"
SELECT table_name
FROM information_schema.tables
WHERE table_schema = 'public'
ORDER BY table_name;
information_schema is a built-in set of tables that describe your database — a catalog Postgres keeps about itself. table_schema = 'public' = "just the normal tables, skip Postgres's internal system stuff."

3. How to read what comes back

You'll see ~12 tables. Mentally sort them into three buckets:

┌──────────────────────────────────────────────────────────────────────────┬──────────────────────────────┬───────────────────────┐
│                                You'll see                                │            Bucket            │      Who made it      │
├──────────────────────────────────────────────────────────────────────────┼──────────────────────────────┼───────────────────────┤
│ django_migrations, django_admin_log, django_session, django_content_type │ Django's own bookkeeping     │ Django, automatically │
├──────────────────────────────────────────────────────────────────────────┼──────────────────────────────┼───────────────────────┤
│ auth_user, auth_group, auth_permission, auth_*                           │ The login/permissions system │ Django's auth app     │
├──────────────────────────────────────────────────────────────────────────┼──────────────────────────────┼───────────────────────┤
│ weather_weatherreading, weather_otpcode                                  │ YOUR project                 │ You wrote these       │
└──────────────────────────────────────────────────────────────────────────┴──────────────────────────────┴───────────────────────┘

The naming rule to notice: Django names every table appname_modelname, lowercase. So your WeatherReading model in the weather app became the table weather_weatherreading. Spot that pattern and you can always find "my" tables in a sea of framework tables — they're the ones prefixed with your app name.

Then — see the actual weather

Once you've spotted weather_weatherreading, peek inside:
SELECT city, temperature, recorded_at, fetched_at
FROM weather_weatherreading
ORDER BY recorded_at
LIMIT 30;
recorded_at = when the weather happens (hourly), fetched_at = the single moment the worker saved them all. Notice every row shares the same fetched_at but different recorded_at — that's your "event time vs ingestion time" distinction, visible in real data. 📊
