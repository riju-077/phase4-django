# NOTES 06 — OTP Email Verification (the wiring)

Prove "this is really you" with a short-lived code. User asks for a code →
we email it → they type it back → we mark them verified.

Builds directly on NOTES_05 (Address → Worker → Display). Nothing new about
the *shape* — this is the same wiring, applied twice.

---

## What OTP is

**One-Time Password** — a 6-digit code that:
1. Works **once**.
2. **Expires fast** (ours: 5 minutes).

Same thing your bank texts you at login. It proves identity for a short window.

---

## The 4-step flow

```
[1] User clicks "Send code"
       → /otp/send/  → send_otp view → emails the code → otp_sent.html
                                        (console = PRINTS in terminal)

[2] User reads code, clicks "Enter the code →"
       → /otp/verify/ (GET) → verify_otp view → shows a blank box → verify_email.html

[3] User types code, hits Verify
       → /otp/verify/ (POST) → verify_otp view → checks code
              correct → is_verified = True → redirect to dashboard ✅
              wrong   → error message, try again
```

The genuinely hard parts were done long before this session. The OTP views
just **use** them:

| Already existed | Where | What it does |
|---|---|---|
| `OTPCode` row auto-created at signup | `signals.py` (`post_save` on User) | every user has one OTP record |

## Worker 1 — send_otp

```python
@login_required
def send_otp(request):
    otp_record = request.user.otp        # the row the SIGNAL auto-made
    code = otp_record.generate_otp()     # MODEL method makes the code

    send_mail(
        subject='Your Dashboard Verification Code',
        message=f'Your OTP code is: {code}\n\nValid for 5 minutes.',
        from_email='noreply@dashboard.com',
        recipient_list=[request.user.email],
    )

    messages.info(request, f'OTP sent to {request.user.email}')
    return render(request, 'weather/otp_sent.html')
```

- `request.user.otp` — reaches the OTP row via the OneToOneField `related_name='otp'`.
- `@login_required` — this view uses `request.user`, so a user MUST be logged in.
  No user → Django redirects to the login page.

## Worker 2 — verify_otp (one view, two jobs)

The key idea: **one view handles both showing the box AND checking the answer**,
decided by `request.method`. This is the standard Django form pattern.

```python
@login_required
def verify_otp(request):
    if request.method == 'POST':          # user SUBMITTED the form
        form = OTPVerifyForm(request.POST)
        if form.is_valid():
            code = form.cleaned_data['code']
            otp_record = request.user.otp

            if otp_record.verify_otp(code):     # pyotp check
                otp_record.is_verified = True   # flip the flag
                otp_record.save()               # write to Postgres
                messages.success(request, 'Email verified successfully!')
                return redirect('dashboard')    # success → go home
            else:
                messages.error(request, 'Invalid or expired code. Try again.')
    else:                                 # first visit (GET)
        form = OTPVerifyForm()            # show a blank box

    return render(request, 'weather/verify_email.html', {'form': form})
```

- **GET** (clicked the link) → make an empty form → show the box.
- **POST** (hit Verify) → validate → check the code → verified or error.

## Why OTPs expire — for free

`verify_otp()` on the model uses pyotp with `interval=300`:

```python
totp = pyotp.TOTP(self.secret, interval=300)   # 300 seconds = 5 min
return totp.verify(code)
```

A code older than 5 minutes makes `verify()` return `False` automatically.
**No timers, no expiry column, no cron.** pyotp math handles it. That's the
whole reason we use a library instead of storing codes ourselves.

Also: we store only the **secret**, never the code. The code is ephemeral —
regenerated each time, never saved.

## The form (forms.py)

```python
class OTPVerifyForm(forms.Form):
    code = forms.CharField(max_length=6, min_length=6, label='Verification Code')
```

One field. `forms.Form` (not `ModelForm`) because we're not saving a model —
just collecting a string to check.

The template renders it with `{{ form.as_p }}` inside a `<form method="post">`
with `{% csrf_token %}` (required on every POST form in Django).

## Is this DRF? NO.

| Feature | Style | Returns |
|---|---|---|
| Chart data (`WeatherChartData`) | **DRF** (`APIView` + `Response`) | JSON numbers |
| OTP flow (`send_otp`, `verify_otp`) | **Plain Django** | full HTML pages |
| Dashboard | Plain Django | full HTML page |

DRF is for the "numbers-only doorway" — when a *machine* (Chart.js) asks for
raw data. OTP is a *human* clicking buttons and reading pages → classic
server-rendered Django (Design A). Right tool for the job.

## Test it end-to-end

1. Be logged in (admin session as `ben` counts).
2. Visit `/otp/send/` → the OTP prints in the **runserver terminal**.
3. Click "Enter the code →" (or visit `/otp/verify/`).
4. Type the 6-digit code → **Verify**.
5. Correct → redirected to dashboard, "verified" message.
6. Confirm ground truth in DBeaver: `SELECT is_verified FROM weather_otpcode;` → `true`.

Wrong or >5-min-old code → "Invalid or expired code" → try again.
