# NOTES 07 — Celery Beat (the bot with a clock)

Make weather fetch itself on a schedule — no human clicking "Refresh."

Builds on the Celery you already had (worker + Redis). Beat adds ONE new thing:
a scheduler that drops jobs into the queue automatically, on a timer.

---

## The why (plain version)

**Before Beat:** new weather only arrives when *you* visit `/fetch/` and trigger
`fetch_weather`. A human has to click. Forever. Even at 3am.

**Beat = an automatic clicker / a bot with a clock.** Its only job:
"every X minutes, press the fetch button for me."

```
You already built:   fetch_weather   (the button / the job)
Beat adds:           a bot that presses it on a timer, unattended
```

Beat does NOT do the work itself. It only *schedules* — it throws the ball,
the worker catches it.

---

## The team (who does what)

```
Beat (alarm clock)  ──rings every 2 min──►  Redis queue  ──►  Worker  ──►  fetch API + save DB
   schedules only                            (the mailbox)      does the actual work
```

- **Beat** = decides WHEN. Sends "due task" into Redis. Never touches the API.
- **Redis** = the mailbox in the middle (same broker you already use).
- **Worker** = decides nothing about timing; just does whatever lands in the queue.

Three roles, cleanly separated. That separation is why production runs them as
separate processes.

## Changes I made — exactly what and why

### 1. `dashboard/settings.py` — added the schedule

```python
from celery.schedules import crontab

CELERY_BEAT_SCHEDULE = {
    'fetch-london-weather': {                     # any name you like (a label)
        'task': 'weather.tasks.fetch_weather',    # WHICH task = the button Beat presses
        'schedule': crontab(minute='*/2'),         # WHEN  = every 2 minutes (demo)
        # 'schedule': crontab(minute=0),           # WHEN  = every hour on the hour (prod)
        'args': ('London', 51.5, -0.1),            # ARGS  = same values you pass to .delay()
    },
}
```

**Why each line:**
- `CELERY_BEAT_SCHEDULE` is a **dict of scheduled jobs.** Add more keys = more jobs.
- `'task'` is the **dotted path** — the same string Celery registered your task under
  (`weather.tasks.fetch_weather`). This is how Beat finds the button.
- `crontab(minute='*/2')` = "at every 2nd minute." Set to `*/2` **only to watch it
  work fast.** Real use = `crontab(minute=0)` = once an hour.
- `'args'` = the arguments handed to the task, exactly like `fetch_weather.delay('London', 51.5, -0.1)`.

**No new package, no migration.** The plain settings-dict scheduler ships with Celery.
(There's an optional upgrade, `django-celery-beat`, that stores schedules in the DB so
you can edit them in the admin — we did NOT use it; the dict is simpler and enough here.)

## Running it (Windows-specific!)

On Mac/Linux you can combine worker + beat in one command (`worker --beat`).
**On Windows that FAILS:**

```
Error: Invalid value for '-B' / '--beat': -B option does not work on Windows.
Please run celery beat as a separate service.
```

So on Windows (and in production everywhere) you run **two separate services**:

**Terminal A — the worker** (catches + does the jobs):
```
celery -A dashboard worker --loglevel=info --pool=solo
```

**Terminal B — beat** (the alarm clock, its own window):
```
celery -A dashboard beat --loglevel=info
```

Plus `python manage.py runserver` in a third terminal = **3 terminals total.**

`--pool=solo` on the worker = the Windows-friendly single-process mode we've been
using since the first Celery run.

---

## How to know it worked

- **Terminal B (beat):** every 2 min → `Scheduler: Sending due task fetch-london-weather`
- **Terminal A (worker):** right after → `Task weather.tasks.fetch_weather... succeeded`

Beat *throws* the ball (B), the worker *catches* it (A). Seeing "Sending due task"
appear with nobody clicking = the milestone. ✅

---

## Watch out

- **It keeps running.** Every 2 minutes it hits Open-Meteo and adds rows. When done
  testing, **Ctrl+C the beat terminal** to stop the clock, or switch to `minute=0`.
- **Worker doesn't auto-reload** on code edits (same as always). Restart it after
  editing `tasks.py`.
- The stale-cache busting still applies: `fetch_weather` calls `cache.delete(...)`,
  so each scheduled run also clears the dashboard/chart sticky-notes. Good — the auto
  data stays fresh on the page too.

## Beat vs OS cron (coming next, chunk 3)

Both are schedulers — easy to confuse:

| | Celery Beat | OS cron |
|---|---|---|
| Lives where | inside Python/Celery | the operating system |
| Runs what | Celery **tasks** | shell **commands** |
| Best for | app jobs (fetch, email) | scripts, backups, `manage.py` commands |

We'll meet cron in chunk 3. Same idea (a timer), different layer.

## The rule to remember

> Beat is just a **bot with a clock**. It doesn't do the work — it *presses a button
> you already built*, on a schedule. Config = one dict entry (task + schedule + args).
> On Windows, run beat and worker as two separate terminals.
