  ---
  📖 The story — "The phone, the language, the translator"

  Django and Redis are on the same laptop. Django CAN'T just talk to Redis yet because:

     ┌───────────────────────┐      ┌──────────────────┐
     │  Django (Python)      │  ?   │  Redis           │
     │  speaks Python        │ ───► │  speaks Redis    │
     │                       │      │  wire protocol   │
     └───────────────────────┘      └──────────────────┘
                    Two parties, different languages.

  We need two libraries between them:

     ┌──────────┐    ┌──────────────┐    ┌─────────────────┐    ┌────────┐
     │ Django   │ ─► │ django-redis │ ─► │ redis (python   │ ─► │ Redis  │
     │ (uses    │    │ (adapter —   │    │ client library) │    │ server │
     │  CACHES) │    │  speaks      │    │ (speaks Redis   │    │        │
     │          │    │  Django's    │    │  wire protocol) │    │        │
     │          │    │  cache API)  │    │                 │    │        │
     └──────────┘    └──────────────┘    └─────────────────┘    └────────┘
                      "Translator"        "Phone line"

  - redis = the Python client (knows how to speak Redis's wire protocol)
  - django-redis = the adapter (plugs into Django's cache framework, uses redis underneath)

  Both needed. They sit at different layers.

  ---
  🧠 Why django-redis (not the simpler built-in option)

  Django 4+ actually ships with a built-in Redis backend (django.core.cache.backends.redis.RedisCache). So why django-redis?

  ┌───────────────────────┬─────────────────────────────────────────────────────────────┐
  │ Built-in Django Redis │                        django-redis                         │
  ├───────────────────────┼─────────────────────────────────────────────────────────────┤
  │ Newer (Django 4+)     │ Older, battle-tested since Django 1.x                       │
  ├───────────────────────┼─────────────────────────────────────────────────────────────┤
  │ Fewer features        │ More features (pub/sub, sentinel, connection pooling, etc.) │
  ├───────────────────────┼─────────────────────────────────────────────────────────────┤
  │ Simpler config        │ More configurable                                           │
  ├───────────────────────┼─────────────────────────────────────────────────────────────┤
  │ OK for basic caching  │ The default for serious Django+Redis projects               │
  └───────────────────────┴─────────────────────────────────────────────────────────────┘

  For Phase 4 — django-redis is the right choice. Most Django+Redis tutorials use it, the docs are excellent, and we'll likely use its richer
  features later.

  ▎ Note: Either works. django-redis is just the community-standard pick.

  ---
  🛠 Step 1 — Install the two packages

  Your project uses uv. The command is:

  uv add redis django-redis

  This will:
  - Add both to pyproject.toml under [project.dependencies]
  - Update uv.lock
  - Install them into .venv

  After it runs, you should see something like:

     + redis==5.x.x
     + django-redis==5.x.x

  ▎ If you want to verify, peek at your updated requirements.txt or pyproject.toml afterwards.

  ---
  🛠 Step 2 — Add the CACHES config to settings.py

  Open D:\DATA_S\django\phase4\dashboard\settings.py and add this block (anywhere reasonable — near the bottom is fine):

  # Cache configuration — uses Redis via django-redis
  CACHES = {
      "default": {
          "BACKEND": "django_redis.cache.RedisCache",
          "LOCATION": "redis://localhost:6379/0",
          "OPTIONS": {
              "CLIENT_CLASS": "django_redis.client.DefaultClient",
          },
      }
  }

  Decoded line by line

  Line: "default"
  Meaning: Django supports multiple named caches. "default" is the one used by cache.get() etc.
  ────────────────────────────────────────
  Line: "BACKEND"
  Meaning: Which class implements the cache. This is the django-redis one.
  ────────────────────────────────────────
  Line: "LOCATION"
  Meaning: Connection URL. localhost:6379 = your container. /0 = Redis logical DB number 0 (Redis has 16 by default).
  ────────────────────────────────────────
  Line: "OPTIONS.CLIENT_CLASS"
  Meaning: django-redis supports multiple "client" strategies (Sentinel, Sharded, etc.). DefaultClient is the standard one.

  ▎ Engineering note (from NOTES_06): the redis:// prefix is the protocol — same pattern as postgres://, mongodb://. Django/django-redis parses
  ▎  this URL to know how to connect.

  ---
  ✅ Step 3 — Verify it works (Django shell)

  After you've installed packages and saved settings.py, open the Django shell:

  python manage.py shell

  Inside the shell:

  from django.core.cache import cache

  # Write
  cache.set('hello', 'phase 4 cache works!', timeout=60)

  # Read
  cache.get('hello')

  Expected output: 'phase 4 cache works!'

  If you get that string back — Django successfully talked to Redis through django-redis and back. 🎉

  Bonus — verify it's actually in Redis

  In another PowerShell:

  docker exec -it redis redis-cli

  Then inside the redis-cli prompt:

  KEYS *

  You should see something like :1:hello (django-redis adds prefix/version). That's PROOF the data went into your Redis container, not some
  in-memory fake.

  Type exit to leave redis-cli.
