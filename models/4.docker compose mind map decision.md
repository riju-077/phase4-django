# NOTES 01 — docker-compose: Postgres + Redis backing services

## The mental model (hotel)
`docker-compose.yml` = the building blueprint + manager. One command (`docker compose up -d`)
opens multiple fully-furnished rooms (services) at once. For now Django still lives on the
host and walks *into* these rooms via published ports.

```
docker compose up -d
   ├── db    (postgres:16)      :5432
   └── redis (redis:7-alpine)   :6379
        ↑ published to 127.0.0.1
   Django runserver (on host) connects to localhost:5432 / :6379
```

## Two core ideas
- **ports: "HOST:CONTAINER"** = the room number visible from the street. We prefix `127.0.0.1:`
  so the service answers to this laptop only, not the whole network.
- **volumes (named)** = a storage closet that outlives the container. Containers are throwaway;
  without `pgdata`, every `docker compose down` wipes the database.

## Reading sources critically (the real skill)
- Official Docker Compose reference  > random blog. When they disagree, official wins.
- **Tell that a tutorial is stale:** it starts with `version: '3.8'`. That field is the obsolete
  Compose v1 marker; modern Compose ignores it and warns. Drop it.
- Blogs hardcoded `POSTGRES_PASSWORD: mysecretpassword` → wrong. Secrets go in `.env`
  (same reasoning as REDIS_URL). Compose auto-reads `.env` for `${...}` substitution.

## Decisions made (and why)
- `postgres:16` NOT alpine → musl libc can bite the DB; play safe on the one stateful service.
- `redis:7-alpine` → alpine is safe for Redis and tiny. Be small where safe, conservative where risky.
- Pin a major version everywhere → parity. Exact number matters less than pinning + consistency.
- `healthcheck: pg_isready` → "container started" ≠ "Postgres accepting connections".
  Pays off later when the web container must wait for a *ready* DB.
