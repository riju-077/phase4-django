OneToOneField vs the ForeignKey you know

You already know ForeignKey from Phase 2 & 3. Let me put them side by side with a picture, because the difference is just "how many."

ForeignKey = many-to-one (what you've always used):

        Post                    User
   ┌──────────────┐
   │ post by akash│ ──────┐
   ├──────────────┤       ├──────►  akash
   │ post by akash│ ──────┤
   ├──────────────┤       │
   │ post by akash│ ──────┘
   └──────────────┘
   MANY posts  →  ONE user

One user can have many posts. That's ForeignKey.

OneToOneField = one-to-one (the new one):

      OTPCode                  User
   ┌──────────────┐
   │ secret for   │ ──────────►  akash
   │ akash        │
   └──────────────┘
   ONE otp  →  ONE user   (and that user can have only ONE otp back)

Each user gets exactly one OTPCode. Never two. The database itself enforces this — try to create a second OTPCode for the same user and it errors. That's the whole point: a user shouldn't have two competing verification secrets.

🧠 Mental shortcut: OneToOneField is just a ForeignKey with a "unique" rule stapled on — "this link can only be used once."

The other piece you'll see is related_name='otp'. That's the reverse shortcut — it lets us write user.otp to get a user's OTPCode back (you used related_name in Phase 3 too). So:

request.user.otp        # → that user's OTPCode row
request.user.otp.secret # → their secret

pyotp — turning a secret + time into a code

Here's the magic, and it's simpler than it looks. The whole thing rests on one shared secret and the current time.

The core idea

   secret  +  current time  ──►  pyotp does math  ──►  6-digit code
  (fixed)      (changes)                                 (482910)

Because the time keeps moving, the code keeps changing. Feed in the same secret 5 minutes later → you get a different code.

Why this is secure (and clever)

The 6-digit code is never stored anywhere. Only the secret lives in your database. The code is computed on demand on both ends:

SERVER (sending)              SERVER (checking)
secret + 12:00  → 482910      secret + 12:02  → 482910  ✅ match, within window
   │                              ▲
   │ emailed to user             │ user types it back
   └──────────────────────────────┘

Same secret + roughly the same time = same code. So the server can verify what the user typed without ever having saved the code. If a hacker steals your database, they get secrets but no live codes — and codes expire in minutes anyway.

The three pyotp calls you'll use

import pyotp

pyotp.random_base32()          # → 'JBSWY3DPEHPK3PXP'  — make a NEW secret (once, per user)

totp = pyotp.TOTP(secret, interval=300)   # build a TOTP engine from a secret
totp.now()                     # → '482910'  — the code RIGHT NOW
totp.verify('482910')          # → True/False — is this code valid right now?

interval=300 = the code stays valid for 300 seconds (5 minutes) before rolling to a new one. That's your "valid for 5 minutes."

🧠 The one-liner to remember: store the secret, never the code. The code is always recalculated from secret + time.
