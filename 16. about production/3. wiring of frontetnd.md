# NOTES 14 — How the frontend and backend are wired (Stitch → Django, and React later)

**Date:** 2026-07-20

I'm not a frontend person. I got my page from **Google Stitch** (AI) and pasted it
into Django. This note answers: **from there on, how does the wiring happen?** —
and how that same wiring would look in a **React** architecture instead.

Companion to [NOTES_13](NOTES_13_architecture.md) (which architecture) and
[NOTES_05](NOTES_05_wiring.md) (the file-to-file flow).

---

## PART 1 — The one truth that unlocks everything

> **Stitch gives me a page full of DEAD buttons.** My only job is to make them
> alive. I never build "frontend" — I just fix the **address** each button points to.

Stitch draws beautiful signs. It writes **fake addresses** on them. Wiring =
replacing the fake address with a real Django one. That's the whole game.

---

## PART 2 — What a button REALLY is

Forget "frontend." In HTML a clickable link is just this:

```
<a href="ADDRESS">  ...pretty stuff...  </a>
```

- `<a>` = "this is clickable."
- `href="ADDRESS"` = **"when clicked, go to this address."**
- Everything *inside* (the `<button>`, icon, colors) = **pure decoration** = the
  Stitch part. It decides how the button *looks*, never where it *goes*.

So a button = **a sign with an address written on it.**

---

## PART 3 — Stitch writes a FAKE address

When Stitch makes a button, it writes:

```
<a href="#">     ← "#" means "go nowhere, stay on this page"
```

A **painted door on a wall.** Looks real, opens onto nothing. Stitch doesn't know my
backend exists — it only draws.

---

## PART 4 — The ONE line I changed (my real Refresh button)

**BEFORE (what Stitch gave me):**
```html
<a href="#">
    <button>Refresh Now</button>
</a>
```

**AFTER (what I wrote):**
```html
<a href="{% url 'trigger_fetch' %}">
    <button>Refresh Now</button>
</a>
```

👉 That `{% url 'trigger_fetch' %}` **is the entire "click → server" wiring.** One
line. I never touched the button, icon, or styling — only the *address on the sign*.

### What that address means

`{% url 'trigger_fetch' %}` says: *"Django, look in your address book (`urls.py`) for
the entry NAMED `trigger_fetch`, and write its real URL here."*

`urls.py` has it:
```python
path('fetch/', views.trigger_fetch, name='trigger_fetch')
#     ↑ the URL       ↑ the worker        ↑ the name the button asked for
```

So Django fills the sign in. The browser actually receives:
```html
<a href="/fetch/">
```

### The click travels

```
I click "Refresh Now"
        │
        ▼
Browser goes to:  /fetch/
        │
        ▼
urls.py matches '/fetch/'  →  calls  views.trigger_fetch
        │
        ▼
the view runs:  fetch_weather.delay('London', 51.5, -0.1)
        │
        ▼
view redirects me back to the dashboard
```

That's the whole thing. A click is just the browser walking to the address I wrote.

## PART 5 — The OTHER half: data going the OTHER way

Buttons send me TO the server. But data also flows FROM the server INTO the page.
Same idea, opposite direction — Stitch left **blanks with fake answers**, I replaced
them with real values.

My real dashboard:
```html
<title>Weather Dashboard - {{ latest.city }}</title>
<h1>{{ latest.city }}</h1>
<span>{{ latest.temperature }}°C</span>

{% for reading in readings %}
    <td>{{ reading.recorded_at|date:"M j, H:i" }}</td>
    <td>{{ reading.temperature }}°C</td>
{% endfor %}
```

- `{{ latest.city }}` = "Django, write the city value here."
- `{% for reading in readings %}` = "Django, repeat this row once per reading."

Where do `latest` and `readings` come from? The **view** hands them over in the
"bag" (the context) — see NOTES_05. Django fills all the blanks **on the server,
BEFORE sending**, so the page arrives already complete.

> **Two directions, one idea:** `{% url %}` = an address going OUT (click → server);
> `{{ value }}` = a value coming IN (server → page). Both are me swapping a Stitch
> fake for a real Django tag. Nothing more.

## PART 6 — How this wiring changes in a REACT architecture

Here I have to **correct one thing** (a super common assumption):

> ❌ **"To switch to React I just change one line — maybe in the .env — and nothing
> else, not even the variables."**
>
> ✅ Reality: switching to React is a **whole different wiring**, not a one-line
> flip. BUT there's a grain of truth hiding in it — let me show both.

### Why it's NOT one line

The two architectures fill the page in **completely different places**:

```
ARCH 1 (Django templates) — filled on the SERVER, before sending
   Django takes the HTML  →  writes {{ city }} = "London" INTO it  →  sends a
   FULL, finished page. The browser just displays it.

ARCH 2 (React) — filled in the BROWSER, after sending
   Django sends an EMPTY page + a pile of JavaScript.
   The JavaScript then asks Django for DATA (JSON), and BUILDS the page live.
```

So the *connection point* moves:

| | Arch 1 (now) | Arch 2 (React) |
|---|---|---|
| Who fills the page? | Django, on the server | React (JS), in the browser |
| Wiring lives in | the **template** (`{{ }}`, `{% url %}`) | **JavaScript** (`fetch(...)`) |
| Django sends | a finished HTML page | only **JSON** (raw data) |
| What I'd rewrite | — | the **whole frontend** as React + views return JSON |

That's a big job: rewrite the frontend in React, and change every view from
`render(template, context)` to returning JSON. **Not one line.**

### The grain of truth (where your instinct is RIGHT)

The **address / connection point** really IS small and swappable. In React, the
"where does my backend live?" value often sits in a single config line, e.g.:

```ini
# React's .env
VITE_API_URL=https://api.mysite.com
```

React reads that and points all its `fetch()` calls at it. So *moving* the backend
(laptop → cloud) can be a one-line change. But that line only says **where** to call
— it is not the wiring itself. The wiring is all the `fetch()` calls + React
components + JSON views around it.

> **So:** the *backend's address* = one config value (your instinct). The *wiring
> between frontend and backend* = a full rewrite (the correction).

### You've ALREADY written React-style wiring once

Look at your **chart**. That one piece is *already* wired the Arch-2 way:

```javascript
fetch('/api/chart-data/')          // JS asks Django for DATA
    .then(r => r.json())           // gets JSON, not a page
    .then(data => new Chart(...))  // JS builds the chart in the browser
```

The rest of your dashboard is Arch 1 (Django fills it). The chart is a little island
of Arch 2 (JS fetches JSON and draws). **Going full React = making the WHOLE page
work like that chart already does.** That's why it's a rewrite, not a line.

## PART 7 — THUMB RULES

1. **A button is a sign with an address.** Stitch writes a fake one (`#`); I rewrite
   it to a real Django name (`{% url 'name' %}`). That's "click → server."
2. **`{{ value }}` pulls data IN; `{% url %}` sends an address OUT.** Same trick,
   two directions.
3. **In Arch 1, Django fills the page on the SERVER before sending.** The page
   arrives finished.
4. **In Arch 2 (React), the BROWSER fills the page after arrival**, using `fetch()`
   to ask Django for JSON. Different place, different wiring.
5. **Switching to React is a rewrite, NOT one line** — the whole frontend + views
   returning JSON. Don't believe the "just change .env" myth.
6. **The backend's ADDRESS can be one config line** (`VITE_API_URL`) — that part of
   your instinct is right. The address is small; the wiring is not.
7. **My chart is already Arch-2 wiring.** Full React = the whole page working like
   the chart does.

---

## One-line summary

> Stitch gives dead buttons and blank fields; I wire them by swapping fakes for
> Django tags — `{% url %}` for clicks going out, `{{ value }}` for data coming in,
> all filled on the server. In a React architecture that wiring moves into the
> browser as `fetch()` calls to JSON endpoints — a full rewrite, **not** a one-line
> change, though the backend's *address* really can live in a single config line.
