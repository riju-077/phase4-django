# NOTES 12 — Deployment (making your site available to the world)

**Date:** 2026-07-20

This note is pure **knowledge / theory** — no code yet. It answers one big question:

> "My project runs on my laptop. How do I make it a real website that
> people around the world can visit and use?"

Read it top to bottom like a story. Each part builds on the one before.

---

## PART 1 — The story: your hotel is stuck in your bedroom

Right now your project runs when you type `runserver` (or `docker compose up`)
on **your laptop**.

Picture it as a **fully-built hotel — but sitting inside your bedroom.**
Everything works: the rooms (containers), the concierge, the kitchen. But:

- The only door is `localhost` / `127.0.0.1`, and that address means
  **"this machine, right here."**
- Nobody else in the world can type `127.0.0.1` and reach *your* hotel.
  On their machine, that same address means *their own* laptop.
- The moment you close your laptop, the hotel vanishes. Lights off.

So to let the world in, you must fix **three** things. This is the whole
heart of deployment:

```
1. A building that's always on   → a computer that never sleeps
2. A real street address         → a public address people can type
3. The hotel actually running    → your app, living on that computer
```

---

## PART 2 — Laptop vs Cloud (and what an IP address is)

### What is an IP address?

An **IP address** is the "phone number" of a computer on the internet — a set of
numbers like `203.0.113.45`. Every machine on a network has one so others can find it.

There are two kinds that matter here:

- **`127.0.0.1` (also called `localhost`)** = a *private* number meaning
  **"me, this very computer."** It is the SAME on every laptop on Earth, and it
  never leaves the machine. Great for testing, useless for the public.
- **A public IP** (like `203.0.113.45`) = a *real, unique* number that anyone on
  the internet can reach.

### Why the laptop can't be the website

| Your laptop | A cloud computer |
|---|---|
| Sleeps, moves around, wifi drops | Always on, always connected |
| Usually has no fixed public IP | Comes with a fixed public IP |
| Turns off → site dead | Runs 24/7 in a data center |

So the fix for "building that's always on" is: **rent a computer in a data center.**
People call it a **server**, a **VM**, or a **cloud instance** — all the same thing:

> **A server = a laptop that lives in a data center and never sleeps.**

You rent it from a **cloud provider** (the landlords): AWS, Google Cloud,
DigitalOcean, Railway, Render, Fly.io, and so on.

### The real street address

Your rented computer comes with a public IP, but nobody wants to type numbers.
So you buy a **domain name** — `myweatherdashboard.com` — and point it at that IP.
Now people type the name, and the internet quietly translates the name → the number
→ finds your building.

(A domain is cheap and *optional at first* — you can test with the raw IP. The
domain is just the pretty sign out front.)

## PART 3 — The two DIALS of deployment

I first said "two ways," but it's cleaner to think of **two separate dials**.
Every deployment is just a setting on each dial.

```
Dial 1 — DELIVERY:    by hand   ⟷   automated (GitHub CI/CD)
                      "How does my code get onto the server?"

Dial 2 — PACKAGING:   raw code  ⟷   Docker image
                      "What do I hand the server — blueprints or a sealed box?"
```

### Dial 1 — Delivery (how the code travels)

```
   WAY A — "by hand" (manual)        WAY B — "conveyor belt" (CI/CD)

  You → copy files to server     You → git push to GitHub
        yourself, every time            ↓
        build there, run there    GitHub tests it
                                        ↓
                                  builds the image
                                        ↓
                                  stores it in ghcr.io
                                        ↓
                                  server pulls it & restarts
```

- **By hand:** you log in, copy the project, build, run. Every code change =
  do the dance again. Simple to grasp, tedious and mistake-prone to repeat.
- **Conveyor belt (CI/CD):** you just `git push`. GitHub tests + builds + stores
  the image; the server grabs the fresh one. Set it up once, then shipping is
  "push to main." This is the CI/CD you've been building in Slice B.

> **Important:** GitHub is NOT what makes the site public. The **rented always-on
> computer** does. GitHub only automates the *delivery*.

### Dial 2 — Packaging (what you ship)

```
WITH Docker:     hand the server a sealed box (image).
                 "Here's the hotel, pre-built. Just plug it in."

WITHOUT Docker:  hand the server your raw code.
                 "Here are the blueprints — YOU build the hotel."
```

Without Docker, the **host builds it for you**: it sees your `pyproject.toml`,
installs Python + your packages, and runs the app with a production server
(gunicorn). Platforms like Render and Railway do this automatically — they call it
a **buildpack**. No `Dockerfile` needed.

> Since you already have a working Docker setup, shipping the **image** keeps your
> dev and prod identical — which was your whole Phase 4 goal. So for *you*, Docker.
> But no-Docker is a completely valid path, especially for beginners.

## PART 4 — The HOSTING LADDER (all the ways to host)

Not project-specific — this is the full map of *where* an app can live.
One idea organizes everything:

> **How much of the work does the landlord do, and how much do you do?**

```
        MORE convenience, LESS control
              ▲
   Rung 4  │  Serverless   — "no building at all"
   Rung 3  │  PaaS         — "landlord builds & runs the hotel"
   Rung 2  │  VPS / IaaS   — "landlord gives an empty building"
   Rung 1  │  Self-host    — "you own the building AND the land"
              ▼
        MORE control, MORE work
```

As you climb: **less control, less work.** That's the eternal trade.

### Rung 1 — Self-hosting (your own hardware)

A computer **you physically own** at home/office — an old laptop, a Raspberry Pi.

- **You own everything:** machine, electricity, internet line, cooling, security.
- Power cut or internet drop → site down. Disk dies → you replace it.
- **Advantages:** cheapest (you own the box), teaches you the most, full control/privacy.
- **Disadvantages:** you are the 24/7 support team; home internet is unreliable
  for a public site.

### Rung 2 — VPS / IaaS ("rent an empty building")

**IaaS** = Infrastructure as a Service. **VPS** = Virtual Private Server.
The "rented computer that never sleeps." Landlord gives an empty building with
electricity + address. Everything inside is yours to install and maintain.

- Examples: DigitalOcean Droplet, AWS EC2, Linode, Hetzner.
- **Advantages:** full control, predictable price, best for *learning what really
  happens*, no vendor lock-in.
- **Disadvantages:** YOU patch the OS, secure it, manage the database — freedom
  *and* responsibility.

### Rung 3 — PaaS ("landlord builds the hotel from your blueprints")

**PaaS** = Platform as a Service. You stop caring about the building. Hand over
your *code*; the platform builds it, runs it, patches the OS, restarts on crash,
and often scales it.

- Examples: Heroku (the original), Render, Railway, Fly.io.
- **This is where the no-Docker "raw code" path lives.**
- **Advantages:** much less work, fast to ship, great for solo devs / small teams.
- **Disadvantages:** less control, you play by *their* rules, can cost more, risk
  of **vendor lock-in** (built around one platform's way of doing things).

### Rung 4 — Serverless ("a room appears only when a guest arrives")

No always-on server in your mind at all. Code sits frozen; a visitor arrives → a
room materializes in milliseconds → serves them → vanishes. Nobody visiting?
Nothing runs, you pay nothing.

- Flavors: **Functions (FaaS)** — tiny code snippets (AWS Lambda, Cloudflare
  Workers); **Serverless containers** — your whole image, scaled to zero when idle
  (Google Cloud Run, AWS Fargate).
- **Advantages:** magical scaling (0 → millions), pay only per use.
- **Disadvantages:** the "room vanishes when idle" model is awkward for things that
  must run **continuously** — like a Celery worker waiting for tasks. Great for
  bursty web traffic, bad for always-listening background jobs.

### Side-branches (not on the main ladder, but you'll meet them)

- **Managed databases (DBaaS):** rent *only* a Postgres or Redis, fully looked-after
  (backups + updates done for you). Examples: AWS RDS, Neon, Supabase, Upstash.
  You'll usually use one of these instead of running your own DB — even on a VPS.
- **Static / frontend hosting:** for pure HTML/JS sites with **no backend**
  (a React or plain landing page). Fast, usually free. Examples: Vercel, Netlify,
  GitHub Pages, Cloudflare Pages. (A Django app can't live here — it needs a running
  backend — but the *frontend half* of many apps does.)
- **Orchestration (Kubernetes / ECS):** for when you have dozens/hundreds of
  containers to auto-manage, heal, and scale — "a company running a whole hotel
  *chain*." Powerful, complex, overkill for one project. Worth knowing the name now.

## PART 5 — "Which host is free for my 4 services?" (the honest answer)

Your app is not one thing — it's **four**:

```
1. Django web        ← most hosts give ONE of these free
2. Celery worker     ← a "background worker"  ⟵ THE PROBLEM
3. Celery beat       ← another background worker
4. Postgres + Redis  ← databases (free tiers are tiny / time-limited)
```

The **background workers** (#2 and #3) are the killer. Free tiers almost always
give *one* free web service and **charge for background workers** — a worker must
run 24/7 waiting for tasks, and hosts can't give that away.

| Host | The catch for a 4-service app |
|---|---|
| **Render** | Free web + free Postgres/Redis (time-limited), but Celery workers are **paid** |
| **Railway** | Runs all 4 easily, but free is a **one-time ~$5 trial credit**, then ~$5/mo |
| **Fly.io** | Small free allowance, runs many machines, but **fiddly** to set up |
| **PythonAnywhere** | Free, but **no Celery, no Redis** — can't run your architecture |

> **Honest bottom line:** a truly free, always-on host for **all 4 services does
> not really exist.** For a demo you can stitch together something free-ish. For a
> real always-on app, budget **~$5/month.**

### Two sane paths

- **Path 1 — "just let people SEE it" (cheapest):** deploy only web + Postgres +
  Redis to a free tier; skip full Celery in the cloud for now. Dashboard + charts
  work; you lose live background fetching. Good for show-and-tell.
- **Path 2 — "the REAL thing" (~$5/mo):** use **Railway**; runs all 4 services
  like your local `docker compose`. Closest to your dev/prod-parity goal.

## PART 6 — THUMB RULES (memorise these)

1. **`localhost` / `127.0.0.1` = "only me."** It can never be your public site.
2. **What makes a site public = a rented always-on computer**, not GitHub.
   GitHub only automates *delivery*.
3. **Two dials, not one choice:** *delivery* (by hand ⟷ CI/CD) and
   *packaging* (raw code ⟷ Docker). Mix freely.
4. **Ladder rule:** want more control → climb DOWN (do more yourself);
   want less hassle → climb UP (let the landlord do it). No rung is "best" —
   only right for a given project + team.
5. **Background workers are what break "free."** Any always-running process
   (Celery) is the thing hosts charge for.
6. **Rent your database, don't run it yourself** (managed DBaaS) unless you have a
   strong reason — backups and updates are done for you.
7. **Serverless ≠ good for always-on jobs.** It's for bursty web traffic that can
   scale to zero, not for a worker that must always listen.
8. **A domain is optional at first** — test on the raw IP, add the pretty name later.

## One-line summary

> Deployment = move the hotel out of your bedroom onto an **always-on rented
> computer** with a **public address**, then choose **how you deliver** the code
> (by hand vs GitHub CI/CD) and **how you package** it (raw code vs Docker image).
> Everything else is just picking a rung on the control-vs-convenience ladder.
