# NOTES 13 — Which architecture am I? (and how to change it)

**Date:** 2026-07-20

A companion to [NOTES_12](NOTES_12_deployment.md). That note was about *where* the
app lives. This one is about *how the app is wired inside* — the shape of the boxes
and who talks to whom.

Pure knowledge again. No code changes.

---

## PART 1 — The answer: I'm on **Architecture 1 (Django Templates)**

There are 4 common web architectures (from simplest to most heavy-duty). Mine is
the simplest one: **#1 — Django serves the HTML itself.**

Here's how I know, checked against my real files:

| The thing that decides the architecture | My project |
|---|---|
| Is there a **React** frontend? | ❌ No — Django builds the pages |
| Is there an **Nginx** traffic-cop in front? | ❌ No |
| Are frontend and backend **separate** services? | ❌ No — one Django app does both |

No React + no Nginx + Django makes the pages = **Architecture 1.**

> This is the same choice we called **"Design A" (SSR = server-side rendering)**
> when I picked Google Stitch. Stitch just gave me the *look*; the HTML is still a
> Django template that Django fills in and sends. That's what makes it Arch 1.

---

## PART 2 — What my wiring actually looks like

The simplest textbook Architecture 1 is **3 boxes**:

```
Django  →  PostgreSQL
        →  Redis
```

But mine has **5 boxes**, because I added Celery (background jobs):

```
   web    (Django + gunicorn — serves pages)  ┐
   worker (Celery — runs background jobs)      ├─ all 3 run the SAME image
   beat   (Celery — the clock/scheduler)      ┘
            │
            ├──► PostgreSQL  (db)      ← stores weather + users
            └──► Redis                 ← broker (jobs) + cache (sticky notes)
```

So I'm **Architecture 1 + background workers.** Adding worker + beat does NOT change
the architecture — it's still "Django serves the pages, no React, no Nginx." It just
adds background muscle.

**Thumb rule:** the architecture is decided by the **frontend + traffic layer**
(React? Nginx?), not by how many backend helpers you bolt on.

---

## PART 3 — Advantages & disadvantages of Architecture 1

### ✅ Advantages

1. **Simplest to build and understand.** One app, one language flow. A request
   comes in → Django makes the page → sends it back. Easy to picture.
2. **Easiest to deploy.** Fewer moving parts = fewer things to break. One web
   service to run, not two.
3. **Perfect for dashboards and admin panels** — exactly what this project is.
   Django's built-in admin + templates were *made* for this.
4. **Django's superpowers come free** — the admin site, forms, template tags,
   auth pages all "just work" because Django owns the whole page.
5. **SEO-friendly by default** — the full page arrives already built, so search
   engines read it easily (React sites have to work harder for this).

### ❌ Disadvantages

1. **Frontend and backend are glued together (tightly coupled).** You can't hand
   the frontend to a separate team, or reuse the same backend for a phone app,
   without extra work.
2. **Whole-page reloads.** Every click that changes data reloads the page. No
   slick "only the ❤️ updates" feel like Instagram (that's React's strength).
3. **Harder to build very rich, app-like interfaces.** Fine for a dashboard,
   limiting for something like Figma or Gmail.
4. **One app does everything** — as the project grows huge, that single app can
   get crowded. (Not a problem at your size.)

> **Bottom line:** for a *data dashboard*, Architecture 1 is not a compromise —
> it's the *right* tool. You'd only outgrow it if you needed a highly interactive,
> app-like frontend or a shared backend for many clients.

## PART 4 — The 4 architectures, and how to climb between them

Think of these as **rungs**, like the hosting ladder in NOTES_12. Each step up adds
a piece and buys you something — at the cost of more complexity.

```
Arch 1  Django serves HTML                    ← YOU ARE HERE
   │    + no React, no Nginx
   ▼
Arch 2  React (separate) + Django as JSON API
   │    two public services, no traffic-cop
   ▼
Arch 3  React + Nginx + Django API            ← "production gold standard"
   │    one public door, backend hidden
   ▼
Arch 4  Load balancer + many backend copies
        for huge traffic (you won't need this for a long time)
```

### Shift from Arch 1 → Arch 2 (add a React frontend)

**What changes:** you stop letting Django build the pages. Instead:
- **React** becomes a *separate* app that builds the pages in the browser.
- **Django stops sending HTML** and starts sending **only JSON** (pure data).
- The browser calls Django's API and React draws the screen.

**What you already have that helps:** your **chart's `/api/chart-data/` endpoint**
is *exactly* this — a Django URL that returns JSON instead of a page. That was the
"doorway to Design B" I pointed at. To go full Arch 2, you'd turn *every* page into
a JSON endpoint like that one, and let React handle all the drawing.

**New things to learn:** React itself, plus a couple of "two separate apps talking"
problems — mainly **CORS** (the browser's rule about one site's JS calling another
site) and **token auth** (login without Django's session cookies).

**You buy:** the slick, no-reload, app-like frontend. **You pay:** two apps to build
and run instead of one.

### Shift from Arch 2 → Arch 3 (add Nginx — the "gold standard")

**The problem Arch 2 has:** *two* public doors (React on one port, Django on
another). The browser can see exactly where your API lives, and two public doors is
messier to secure and manage.

**What Nginx fixes:** Nginx is a **traffic-cop** (a *reverse proxy*) that sits in
front as the **single public door**. It reads the address and routes:

```
Browser → Nginx → "/"        → React
                → "/api/*"    → Django
```

Now there's **one** public entry point, the backend is **hidden** behind Nginx, and
routing is clean. This is what most real production apps look like.

**You buy:** one secure door, hidden backend, easier to scale. **You pay:** learning
Nginx config.

### Shift from Arch 3 → Arch 4 (load balancer, high traffic)

Only when **one Django box can't keep up** with the number of visitors. You run
*several copies* of the backend and put a **load balancer** in front to spread
requests across them.

```
Load Balancer → Backend #1
             → Backend #2   (all share the same PostgreSQL + Redis)
```

**You won't need this for a long time.** Note it, don't build it.

> **Golden thumb rule for climbing:** only climb a rung when a *real pain* pushes
> you — reloads feel clunky (→ Arch 2), two doors feel messy (→ Arch 3), one box
> can't cope (→ Arch 4). **Never climb for the résumé.** Complexity you don't need
> is just weight you have to carry.

## PART 5 — How to make THIS wiring (Arch 1) available to the world

Good news: **you do NOT need to change architecture to go public.** You just wrap
your existing 5 boxes in the deployment layer from NOTES_12. Here's the shape:

```
                    Browser (anyone, anywhere)
                            │
                            ▼
                     Cloudflare            ← optional gatekeeper:
                 (SSL, DNS, CDN, DDoS)        HTTPS + a pretty domain + protection
                            │
                            ▼
                  VPS / rented computer     ← the "always-on laptop" from NOTES_12
                            │
                     Docker Compose         ← your SAME docker-compose.yml
                            │
        ┌───────────────────┼────────────────────┐
        ▼         ▼          ▼          ▼          ▼
      web      worker      beat        db        redis
   (public)  (internal)  (internal) (internal) (internal)
```

### What actually has to change (small list)

1. **Rent a computer (VPS)** and install Docker on it — from NOTES_12.
2. **Get your image / code onto it** — either `git clone` + `docker compose up`
   (by hand) or the ghcr.io conveyor belt (CI/CD). Both from NOTES_12.
3. **Open the web door to the world.** Right now your compose says:
   ```
   ports:
     - "127.0.0.1:8000:8000"   ← "only my laptop can reach it"
   ```
   On the server, `web` must accept public traffic instead of localhost-only.
   **Leave db + redis + worker + beat internal** — they should NEVER be public.
   (That's the `ports:` vs `expose:` idea from temp.txt: only the front door gets
   `ports:` to the outside; the rest stay on Docker's private network.)
4. **Fix the production settings** (mostly already done!):
   - `DEBUG=False` ✅ (already the container default)
   - `ALLOWED_HOSTS` must include your real domain ✅ (already reads from config)
   - a fresh, secret `SECRET_KEY` ⚠️ (still the insecure dev key — regenerate)
   - static files via whitenoise ✅ (already done)
5. **(Optional) Point Cloudflare + a domain at the server** for HTTPS and a nice
   name — the "pretty sign over the IP" from NOTES_12.

### The key insight

> **Architecture** (how the boxes are wired) and **deployment** (where they run and
> who can reach them) are **two separate switches.** Going public flips the
> *deployment* switch. Your *architecture* stays Arch 1 the whole time. You are not
> rebuilding — you're relocating.

## PART 6 — THUMB RULES (memorise these)

1. **Architecture is decided by the front layer:** React? Nginx? If neither, and
   Django makes the pages → you're **Architecture 1.**
2. **Backend helpers (Celery, extra DBs) don't change the architecture** — they
   just add muscle. 3 boxes or 5 boxes, still Arch 1.
3. **Arch 1 is the RIGHT choice for dashboards and admin panels** — not a beginner
   compromise. Match the tool to the job.
4. **Climb rungs only under real pain**, never for the résumé:
   reloads clunky → Arch 2; two public doors messy → Arch 3; one box maxed → Arch 4.
5. **Your chart's JSON endpoint is the seed of Arch 2** — the door to React is
   already cracked open; you just haven't walked through it.
6. **Architecture ≠ deployment.** You can go fully public *without* changing your
   architecture. Relocate, don't rebuild.
7. **Only the front door gets `ports:` to the world.** Database, Redis, and
   background workers stay internal — always.

## One-line summary

> I'm on **Architecture 1 (Django serves the HTML) + Celery workers** — the simplest,
> right-sized shape for a dashboard. To go public I keep this exact wiring and just
> add the NOTES_12 deployment layer (VPS + open the web door + optional Cloudflare).
> To grow into React someday I'd climb to Arch 2 (JSON API + React), then Arch 3
> (add Nginx) — but only if a real need pushes me there.
