# NOTES 05 — Wiring: urls.py → views.py → template

The thing that hurt across Phase 1–3: three files, constant jumping between
them, losing the thread. This is the fix. Learn the *shape* once and the
file-jumping stops feeling like jumping.

---

## The one-way street

A web request only ever flows in ONE direction. It never loops back.

```
Browser  ──►  urls.py  ──►  views.py  ──►  (DB / cache / task)  ──►  template  ──►  Browser
 (asks)      (address)     (worker)         (fetch stuff)          (display)      (sees HTML)
```

You read it left to right, every single time. No exceptions.

---

## The 3 roles — "Address → Worker → Display"

| File | Role | Hotel metaphor | The one question it answers |
|---|---|---|---|
| `urls.py` | **Address** | The front desk directory | "Which room does this URL go to?" |
| `views.py` | **Worker** | The staff member in the room | "What work do I do, and what do I hand back?" |
| `template` | **Display** | The room as the guest sees it | "How is the finished result shown?" |

When you're lost, ask which of these 3 jobs you're doing *right now*. You are
only ever in one of them.

---

## Build order (always the same)

1. **Address first** — add a `path()` in `urls.py`. Decide the URL and the name.
2. **Worker next** — write the view function. Get the data, pack the `context` bag.
3. **Display last** — write the template that unpacks the bag.

You build in the same direction the request flows. That's why it stops feeling
like jumping — you're walking downstream.

---

## The "bag" — how the worker hands off to the display

The view packs a dict called `context`. The template unpacks it by key name.

```python
# views.py — pack the bag
context = {
    'latest': latest,        # key 'latest'  → template uses {{ latest }}
    'readings': readings,    # key 'readings'→ template loops {% for r in readings %}
}
return render(request, 'weather/dashboard.html', context)
```

```html
<!-- dashboard.html — unpack the bag by the SAME key names -->
{{ latest.temperature }}
{% for r in readings %} {{ r.city }} {% endfor %}
```

The key strings are the contract between worker and display. Match them exactly.

## Our actual dashboard, mapped onto the model

```
GET '/'                     ← Browser asks
  → urls.py: path('', views.dashboard, name='dashboard')   ← Address
  → views.py: dashboard()                                  ← Worker
        latest   = WeatherReading...first()   (headline row)
        readings = WeatherReading...[:24]     (table rows)
        context  = {latest, readings}         (pack the bag)
  → dashboard.html                            ← Display
  → Browser sees real London weather
```

---

## The rule to remember

> **Address → Worker → Display.** Build in that order, read in that order.
> The request only flows one way, so you never have to hold two files in your
> head at once — finish one role, walk downstream to the next.
