Yes ✅ — in Design B the frontend person and backend person work separately, in their own apps. Less day-to-day tangling. True.

But the wiring doesn't vanish — it moves. In B, the two sides must agree on the data. If the backend sends temp but the frontend expects temperature, the chart breaks. So instead of tangling inside one file, they now depend on a handshake: "we both agreed the data looks like this." Break the handshake, break the app.

So: B isn't "no wiring." It's "wiring happens at the handshake between two apps" instead of "wiring happens inside one page." Different spot, not gone.

Now — the Design A wiring, and your thumb rule 🔑

When Stitch gives you a page, it's full of fake data it made up to look pretty. Like:
<h1>London</h1>
<p>14°C</p>
That "London" and "14°C" are fake — Stitch invented them. Your whole wiring job in A is one thing:

▎ 🔑 THUMB RULE
▎
▎ Hunt down every fake value Stitch invented, and replace it with a Django tag that pulls the real one.

Two kinds of replacements, that's it:

1. A single fake value → {{ }}
<p>14°C</p>          →     <p>{{ temperature }}°C</p>
      fake                          real, from your database

2. A fake repeated list → {% for %} loop
If Stitch shows 3 fake weather rows, you wrap one row in a loop so Django repeats it for all your real rows:
{% for reading in readings %}
    <p>{{ reading.temperature }}°C</p>
{% endfor %}

Why it feels complex (but isn't)

In A, the data and the page live mixed in one file. So the "hard" part is just spotting which bits are fake data vs which bits are design. Once you can spot the fake values, wiring is just find-and-replace. That's the whole skill.

Design A wiring — the steps

Step 1 — Find the blanks.
Read the page. Mark every value that should be real data, not fake.
→ You mark London and 14°C.

Step 2 — Name each blank.
Give each a simple name.
→ London = city. 14°C = temperature.

Step 3 — Ask: one value, or a list?
For each blank, is it a single value, or many that repeat?
→ city = one. temperature over 24 hours = a list (many).

Step 4 — Turn them into blanks.
- One value → {{ city }}
- A repeating list → a {% for %} loop that repeats a row

So the page becomes:
{{ city }}
{{ temperature }}°C

Step 5 — Backend fills the blanks.
In your view, fetch the real data and hand it to the page:
→ "city = London, temperature = 14" → pass to the template.

Step 6 — Load the page.
Django swaps each blank for the real value. Done. The user sees real weather.

In one line: find blanks → name them → single or list? → make them {{ }} or {% for %} → backend feeds them.

Now — Design B conflicts (I didn't skip you)

In B, the two apps agree on a "data shape." A conflict happens when someone breaks that agreement.

Example:
- They agreed: the data has a field named temp.
- Later, the backend dev renames temp → temperature.
- The frontend still looks for temp → finds nothing → chart breaks.

Nobody was editing the same file, so nobody noticed until it broke in front of users. That's the B conflict — a broken handshake between two apps that don't talk directly.

A vs B, the real trade:
- A: everything's in one file → messier, but you see it all in one place.
- B: clean separation → but the two sides can silently drift apart and break the handshake.
